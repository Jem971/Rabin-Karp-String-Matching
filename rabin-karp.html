What it does
Given a long text and a short pattern, Rabin-Karp finds every position where the pattern appears — like Ctrl+F in a document.

The core idea
Rabin-Karp converts each window of text into a single number (a hash) and compares that number against the pattern's hash. Most windows won't match, so they're skipped instantly without touching individual characters. Only on a hash match does it verify the actual letters. The key trick is that as the window slides, the hash is updated in one step — dropping the old character and adding the new one — rather than recomputed from scratch.

How it works
1️⃣Hash the pattern once.
2️⃣Hash the first window.
3️⃣If hashes match, verify characters.
4️⃣Slide the window using the rolling hash trick.
5️⃣Repeat.
Hash function
H = Σ char[i] × 256(m−1−i) mod 101
Each character has an ASCII value and is weighted by its position — leftmost characters carry the highest power of BASE (256). Taking mod 101 (a prime) keeps the number small while minimizing collisions.
Rolling hash update
H_new = (H_old − out × 256m−1) × 256 + in (mod 101)
Remove the outgoing character's contribution, shift remaining chars left (×BASE), then add the new character — all in O(1) without touching the middle characters.

Common Mistakes
⚠️ Skipping Verification
Hash equal ≠ strings equal. Always confirm with a character check after a hash match.
⚠️ Bad MOD Choice
If MOD is too small, almost every window collides → O(n·m). Always use a large prime.

Why Rolling Hash Matters
Without rolling hash: Re-hash all m characters per step → O(m) × n steps = O(n·m) always.

With rolling hash: Each slide is O(1) (subtract one char, add one char). Total = O(n) → O(n+m) average.

Core Idea
Make the locally best decision at each step without reconsidering.
Greedy Choices in Rabin-Karp
At every window position, three irrevocable greedy decisions are made:
1
Hash mismatch → skip immediately. Trust the hash, never look back.
2
Hash match → verify now. Minimum work: only compare chars when forced.
3
Roll hash greedily. Reuse previous result — never redo work.

❌ What Goes Wrong Without Greedy (Exhaustive approach)
Imagine checking every character at every position, even after a hash mismatch:
text = "banana", pattern = "ana" Position 0: b≠a → STOP (greedy skips rest ✓) WITHOUT greedy: compare a,n,a anyway → 2 extra ops per window Extra cost: (n-m+1) × m = 4×3 = 12 ops instead of ~4
Exhaustive: O(n·m) always, even on easy inputs
Greedy (Rabin-Karp): O(n+m) average — skip immediately on hash mismatch
Where Greedy Breaks Down
If MOD is too small (e.g. MOD=3), many windows hash identically. The skip decision can't be trusted → performance collapses toward O(n·m). The greedy approach is only as good as the hash function it relies on.

Dynamic Programming
Core Idea
Solve overlapping subproblems once, store results. Examples: LCS, Edit Distance, KMP failure function.
DP Elements in Rabin-Karp
The rolling hash is a DP recurrence: each hash is computed from the previous → store and build on it.

KMP goes further with a proper DP failure table in O(m), guaranteeing O(n+m) worst case.

Divide & Conquer
Core Idea
Split recursively into independent subproblems, solve each, combine results. Examples: Merge Sort, Quick Sort, Binary Search.
Does Rabin-Karp Use D&C?
Not directly. Rabin-Karp is purely iterative. D&C on string search gives O(n log n) — worse than O(n+m) average.
❌ D&C Gives Wrong Answer at Boundaries
Split "abcabc" in half, search each side for pattern "ca":
Left: "abc" → no "ca" found Right: "abc" → no "ca" found Result: 0 matches — WRONG! Actual: "ca" at index 2 spans the split boundary
D&C misses any match that crosses the midpoint. You need extra boundary logic, making it both harder to implement and slower (O(n log n)).

Brute Force algorithm:
For each position i (0 to n−m), compare text[i..i+m-1] with pattern character by character. If all m chars match, report a match. Move to i+1. Cost: O(n·m) always.

Rabin-Karp:
Compute a hash for each window. Only when the hash equals the pattern hash do we do the O(m) char comparison. With a good hash, collisions are rare → most windows skip char comparison entirely. Cost: O(n+m) average.



📊 Concrete numbers — text: "banana bandana", pattern: "ana"
Brute force: (14−3+1) × 3 = 36 character comparisons in worst case.
Rabin-Karp: 12 hash comparisons + char checks only on hash matches ≈ ~15–18 total operations.
Rabin-Karp saves ~50% of the work on this example.
